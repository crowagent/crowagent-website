### Compression + format gap (documented per H3-HERO-VIDEO spec)

The build environment used to ship this asset (Node 24, Windows) does
**not** have `ffmpeg` available on PATH and no `ffmpeg-static` /
`@ffmpeg-installer/ffmpeg` package is installed in
`crowagent-website/node_modules`. As a result:

- The 720p MP4 is shipped **as downloaded** from Mixkit's CDN
  (`assets.mixkit.co/videos/51105/51105-720.mp4`). No re-encode at CRF
  28 was performed; bitrate is whatever Mixkit shipped (~1.94 MiB /
  ~1.99 MB for the 6-second loop, just under the 2 MiB cap; ~2.03 MB
  on a base-10 reading, which the spec carve-out anticipates: "If
  video size cannot reach <2 MB, document gap + use highest available
  compression.").
- A **WebM (VP9) sibling was NOT generated** — the `<source>` block in
  `index.html` lists `hero-loop.webm` first so the browser will fall
  through to the MP4 today and pick up the WebM the moment a future
  agent (with ffmpeg) drops it next to the MP4.
- The 360p MP4 (433 KiB / ~423 KB) is included as a save-data fallback
  for low-bandwidth users and as the lower-DPI source that a future
  `<source media="...">` rule can prefer.

Action item for the next agent that gains ffmpeg: re-encode
`hero-source.mp4` to H.264 at CRF 28 + VP9 WebM, both ≤2 MB, place
beside the existing files. The HTML already references both filenames.
