## Section B — Manual keyboard navigation findings

### B-1 Keyboard trap in cookie banner (WCAG 2.1.2 Level A — **critical, blocks every page**)
**Evidence:** Loaded `/` cold, pressed Tab 25 times. Focus cycles only across:
`#ca-cookie-manage` → `#ca-cookie-reject` → `#ca-cookie-accept` → `a (Cookie policy)` → loop. The skip-link `<a href="#main-content">Skip to main content</a>` is never reached. Keyboard-only users cannot dismiss the banner without making a privacy decision and cannot reach page content.

**Source:** `js/cookie-banner.js` lines 230-260 (banner DOM) — no `focus()` trap escape; banner is appended after the skip link in DOM order but tabindex/focus-management is missing. Skip link is **physically before** the banner in DOM, so the cookie banner must be intercepting via JS or appending its container with a higher z-index that captures focus.

### B-2 No visible page-content focus indicator on first Tab
Once past the cookie banner (after accepting), the global skip link (`a[href="#main-content"]`) exists per probe but is `position: absolute; top: -40px` (default sr-only); it must reveal on `:focus` — needs runtime verification once trap is fixed.

### B-3 Focus ring on `#ca-cookie-accept` is invisible
Computed `outline: solid 3px rgb(4, 14, 26)` (= `--bg-dark` = #040e1a) on a teal `var(--teal)` background. Ratio ≈ 1.5:1. WCAG 2.4.11 (Focus Appearance, WCAG 2.2 AA) requires the focus indicator to have ≥3:1 contrast with the unfocused state. **File:** `styles.css:1901` (`.btn-cookie-primary`) — focus ring inherits page-level outline `var(--bg)`.

### B-4 Nav-dropdown chevrons are 10×44 px
`<span class="nav-dropdown-chevron" role="button" tabindex="0" aria-label="Open Products menu">` measures **10 px wide**. WCAG 2.2 SC 2.5.8 requires ≥24×24 px (or exception via spacing). axe reports this on **every page** (`js/nav-inject.js` line ~140-180 injects this markup; CSS at `styles.css` for `.nav-dropdown-chevron`).

## Section C — Contrast failures (with hex + ratio)

axe reports **114 contrast nodes** across the 10 pages. The dominant root cause is a **light-mode token leak** (not 114 unique defects).

### C-1 Root-cause analysis — light-mode CSS variable leak (affects ~95% of contrast nodes)

In `prefers-color-scheme: light` (Chromium default headless), the site renders body bg = `#F7F9FC` (`--bg-light`) **but** the cascade pins `--cloud` to `#E8F0FA` (the **dark-mode** ink) on body-scoped overrides:

| File | Line | Selector | Effect |
|---|---|---|---|
| `styles.css` | 30237-30269 | `body.f8-product { --cloud: var(--cloud-dark) !important; ... }` | Pins ink to dark colour on product pages |
| `styles.css` | 30293-30334 | `body.f8-pricing, body.f8-roadmap, body.f8-faq, body.f8-changelog, body.f8-resources, body.f8-products, body.f8-tools-index, body.f8-glossary, body.blog-index-page, body.f8-404 { --cloud: var(--cloud-dark) !important; ... }` | Same defect on marketing pages |
| `styles.css` | 32230-32234 | `html, body { background: var(--bg-dark) !important; color: var(--cloud-dark) !important }` | Pins body bg to dark — but ONLY this declaration loses to the later `@media (light)` block at line 32272, while the body-class scope above (higher specificity than `html, body`) wins on `--cloud` |
| `styles.css` | 32251-32334 | `@media (prefers-color-scheme: light) { :root { --cloud: var(--cloud-light) ... }; body { background: var(--bg-light) ... } }` | Switches `:root` and `body bg` to light, but **does not override `body.f8-product` / `body.f8-pricing` etc. `--cloud` overrides above** (lower specificity loses) |

**Net result under light mode:** body bg = `#F7F9FC` (light) ; H2/breadcrumb/body text resolves `var(--cloud)` to `#E8F0FA` (dark-mode pale blue). Ratio ≈ **1.08:1** — total invisibility. CDP verification: `getComputedStyle(document.documentElement).getPropertyValue('--cloud')` → `#0B1828`; `getComputedStyle(document.body).getPropertyValue('--cloud')` → `#E8F0FA`. The body-class rule overrides root.

**This matches the known incident in MEMORY (`feedback_no_agent_trust_pixel_verify` 2026-05-23).** Light-mode leak is NOT fixed.

### Is the site WCAG 2.2 AA conformant today? — **NO.**

**Reasoning (3 hard blockers):**
1. **Keyboard trap in the cookie banner** (SC 2.1.2 Level A) — every keyboard user is unable to reach page content on every page. A single Level A failure invalidates AA conformance.
2. **Body-text contrast 1.08:1 under `prefers-color-scheme: light`** (SC 1.4.3 AA) — affects 95% of headings and body copy on the 10 audited pages. Root cause is the `body.f8-product` / `body.f8-*` `--cloud` !important pin at `styles.css:30237-30334` losing the cascade race with the light-mode `:root` remap that ships the page bg.
3. **Touch-target size 10×24 px nav chevrons** (SC 2.5.8 WCAG 2.2 AA) — present on every page, no spacing exception applies.

Fixes #1, #2, #3 in the table above unblock conformance for ~95% of the surface. The remaining 12 defects are scoped and addressable in <1 day of focused work.
